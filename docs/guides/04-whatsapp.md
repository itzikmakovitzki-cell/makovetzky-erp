# WhatsApp ב-ERP

המערכת מתחברת ל-WhatsApp דרך **Green API** (חשבון לקוח) לשלושה תפקידים:

1. **קליטת מסמכים** מהשטח (תיבת WhatsApp).
2. **קבוצות פרויקט** — שיוך קבוצה ל-MasterDeal, קליטת הודעות לציר הזמן.
3. **שליחה יוצאת ידנית** — תזכורות, איפוס סיסמה, עדכון ללקוח.

> 🛡️ **חוק ברזל:** אין שום cron / scheduler / queue ששולח אוטומטית. כל שליחה היא תוצאה של לחיצה ידנית של אדמין בתוך המערכת. בלקוחות רגישים זה קריטי.

## חלק א׳: תיבת WhatsApp — `/inbox`

### איך מסמכים נכנסים

הוובהוק ב-`/api/whatsapp/webhook` (Meta / Green API) או הוובהוק הגנרי ב-`/api/webhooks/whatsapp` מקבל הודעות נכנסות. כל מסמך נשמר כ-`PendingDocument` עם:

- שם קובץ
- URL ב-Supabase Storage
- מקור (`WHATSAPP` / `EMAIL` / ...)
- פרטי שולח
- ההודעה הגולמית
- סטטוס (`PENDING` / `ASSIGNED` / `REJECTED`)

ה-PendingDocument **לא משויך עדיין להיתר** — זאת ה-Triage שלכם.

### תהליך Triage

ב-`/inbox` רואים את כל ה-PendingDocuments הממתינים. לכל אחד שתי פעולות:

**שייך** — פותח דיאלוג שבו בוחרים:
- פרויקט (MasterDeal)
- היתר (Permit) — מסונן אוטומטית לפי הפרויקט
- משימה (Task) או בניין (Building) — אופציונלי
- אישור ע"י אדמין (checkbox)

לחיצה על "שייך" יוצרת `Document` רשמי תחת ההיתר ועוברת את ה-`PendingDocument` ל-`ASSIGNED`.

**דחה** — דורש סיבה חופשית. ה-`PendingDocument` עובר ל-`REJECTED` עם הסיבה. עתידית תישלח חזרה לשולח דרך הבוט.

### במובייל

תיבת ה-WhatsApp הייתה הראשונה שעברה לכרטיסי מובייל — היא בשימוש יומיומי. כל מסמך הוא כרטיס עם תצוגה מקדימה (תמונה לחיצה לפתיחה), מקור, שולח, סטטוס, ולחצני פעולה.

## חלק ב׳: קבוצות פרויקט (WhatsApp Groups)

נוסף ב-PR-1 → PR-3 + PR #60. הרעיון: לחבר קבוצת WhatsApp של פרויקט ל-MasterDeal במערכת, כך שכל הודעה בקבוצה שתויגה את הבוט (`@mention`) או ציטטה הודעה שלו תיכנס לציר הזמן של הפרויקט.

### מודל הנתונים

`ProjectWhatsAppGroup` עם:
- `masterDealId` (nullable + unique — מאפשר orphans)
- `groupChatId` (לדוגמה `972...@g.us`)
- `groupName` (cache מהוובהוק)
- `connectedAt`, `connectedById`
- `isActive`

### זרימת חיבור

1. הוסיפו את הבוט (Green API instance) לקבוצת WhatsApp של הפרויקט.
2. ההודעה הראשונה שמתקבלת מהקבוצה יוצרת רשומת **Orphan** — קיים `groupChatId` אבל אין `masterDealId`.
3. ב-`/inbox`, מעל תיבת המסמכים, יש פאנל **"קבוצות לא משויכות"** עם הקבוצה החדשה.
4. אדמין לוחץ "שייך לפרויקט" → בוחר MasterDeal → הקבוצה מקושרת.

### קליטת הודעות

הודעות שהבוט נתייג בהן (`@mention`) או הודעות שמצטטות את הבוט נשמרות כ-`PendingDocument` עם `groupChatId`, `authorName`, `authorPhone`, ואופציונלית `suggestedTaskName`.

### ציר זמן בפרויקט

ב-`/projects/[id]/whatsapp` יש tab ייעודי שמציג את כל ההודעות מהקבוצה שנקלטו (chronological, עם author, message preview, ולינק drill-down).

## חלק ג׳: שליחה יוצאת ידנית

### תזכורת למבצע משימה

על כל משימה (בכל מסך שמראה את האחראי) יש כפתור 📱 בצבע ירוק WhatsApp. לחיצה:

1. מעתיקה ל-clipboard הודעה מוכנה:  
   `"היי [שם], תזכורת לגבי משימה: [משימה] בפרויקט [פרויקט]. אפשר סטטוס?"`
2. פותחת `wa.me/[phone]?text=...` בכרטיסייה חדשה (רק אם לאחראי יש מספר טלפון).

זאת הצורה הקלאסית. ב-Green API יש גם נתיב ישיר — ראו "שליחה ללקוח" למטה.

### שליחה ללקוח דרך פרופיל הלקוח

ב-`/clients/[id]` יש פאנל **WhatsApp**:

1. **פיל מצב התקשרות** — `🔕 כבוי` / `✋ ידני בלבד`. אם כבוי, ה-UI לשליחה **נמחק מה-DOM** (לא רק מנוטרל) — אי אפשר לעשות מיסקליק לסכוןאד.
2. אם ידני, יש כפתור **"שלח עדכון"** שפותח dialog עם textarea ניתן לעריכה.
3. כפתור השליחה משתנה לפי הטרנספורט:
   - **Green API מוגדר** — `📤 שלח עכשיו`. דורש `window.confirm` נוסף לפני שליחה ("לשלוח את ההודעה הזו ל-X?"). פרסום ישיר דרך השרת.
   - **לא מוגדר** — `פתח WhatsApp`. נופל ל-wa.me, האדמין לוחץ שלח ב-WhatsApp עצמו.

### Audit Log לכל שליחה

כל שליחה מתועדת ב-AuditLog עם אחת מ-3 פעולות:
- `whatsapp_sent` — Green API הצליח, כולל `idMessage`
- `whatsapp_send_failed` — Green API נפל, מצוטט שגיאה ונפילה חזרה ל-wa.me
- `whatsapp_opened` — wa.me נפתח (fallback או לא מוגדר)

### תצורת Green API

שני env vars שצריכים להיות מוגדרים ב-Vercel לשליחה ישירה:

```
GREEN_API_ID_INSTANCE
GREEN_API_TOKEN_INSTANCE
```

ניתן להוציא מ-[console.green-api.com](https://console.green-api.com/). אם לא מוגדרים, המערכת חוזרת אוטומטית ל-wa.me.

## חלק ד׳: איפוס סיסמה דרך WhatsApp (Block 28)

המשתמש שכח סיסמה? אין `/forgot-password` ידני — הוא ציבורי לחלוטין.

### זרימה

1. ב-`/forgot-password` המשתמש מכניס את **מספר הטלפון** שלו (לא אימייל).
2. השרת מחפש משתמש עם הטלפון הזה ב-DB.
3. **אם נמצא** — נשלח `PasswordResetToken` (32 תווים, תוקף שעה, חד-פעמי) דרך Green API לטלפון הזה:
   `"קישור לאיפוס סיסמה: [URL]/reset-password/[token]"`
4. **אם לא נמצא** — מוצגת **אותה תגובה** ("בדוק את ה-WhatsApp שלך"). הגנה מ-account enumeration.

### דף איפוס

`/reset-password/[token]` בודק:
- הטוקן קיים
- לא פג תוקף
- לא בשימוש

אם תקין — שדות סיסמה חדשה + אישור → שמירה → ה-`Token.usedAt` מסומן → המשתמש מועבר ל-`/login`.

### דרישות

- `SYSTEM_WHATSAPP_PHONE` ו-`SYSTEM_WHATSAPP_NAME` מוגדרים ב-Vercel (שולח הוובהוק).
- `NEXT_PUBLIC_APP_URL` — חיוני! ההודעה משתמשת בו לבניית ה-URL. אם לא מוגדר, נופל ל-`VERCEL_URL` עם דומיין `.vercel.app`.

## טיפים שיחסכו לך כאב ראש

- **הקפד שלכל משתמש יש טלפון** ב-`/settings/users`. בלעדיו, כפתורי WhatsApp לא יעבדו ועלולים להציג "טרם הוזן".
- **לקוחות רגישים** — סמן OFF בפרופיל הלקוח. ה-UI לשליחה לא יוצג בכלל, לא רק מנוטרל.
- **Green API mass-send אסור** — המערכת תוכננה במכוון בלי scheduler. אל תוסיף cron שעוקף את זה.
- **בדוק את ה-AuditLog** אחרי שליחה — `whatsapp_sent` מאשר שעבר, `whatsapp_send_failed` מסביר למה נכשל.
- **קבוצות orphans** — אם הקבוצה כבר לא רלוונטית, אפשר לדחות אותה מ-/inbox, או להשאיר אם תתקבל הודעה חדשה בעתיד.
